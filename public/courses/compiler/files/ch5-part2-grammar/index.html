Formal Definition

A grammar G = (V_N, V_T, S, P) where:

  1. V_N — a finite set of non-terminals.
  2. V_T — a finite set of terminals.
  3. S ∈ V_N — the starting symbol of the grammar.
  4. P — a set of production rules that define the grammar structure.

Notes:

  • V_N ∩ V_T = ∅ — the two sets are disjoint.
  • V_N ∪ V_T = V — together they form the full vocabulary.

Notation conventions used throughout this chapter:

  • Uppercase letters A, B, …, Z → non-terminals
  • Lowercase letters a, b, …, z → terminals
  • Greek letters α, β, γ, … → strings formed from V_N or V_T (i.e. from V)

For example, if V_N = {S, A, B} and V_T = {0, 1}:

α = A11B
β = S110B
γ = 0010

Productions

A production α → β (read “alpha derives beta”) is a rewriting rule that allows us to replace any occurrence of α with β in a string.

The left-hand side α must contain at least one non-terminal from V_N.

If we have the string γασ, we can apply the production α → β to obtain γβσ.

A derivation is a sequence of strings obtained by repeatedly applying productions:

α₀ → α₁ → α₂ → ... → αn

Given a grammar G, L(G) is the set of all strings (the language) that can be generated by G.

Example 1:

G = ({S, B, C}, {a, b, c}, S, P)

Productions:
  S → aSBC
  S → abC
  CB → BC
  bB → bb
  bC → bc
  cC → cc

Some derivations:

S → abC → abc                           ∈ L(G)
S → aSBC → aabCBC → aabBCC → aabbCC → aabbcC → aabbcc   ∈ L(G)
S → aSBC → ... → aaabbbccc              ∈ L(G)

Therefore: L(G) = { aⁿbⁿcⁿ | n ≥ 1 }

There is no exact method to find L(G) — the idea is to try different derivation paths and identify the pattern.

Example 2:

G = ({E, T, F}, {+, *, (, ), n}, E, P)

Productions:
  E → E + T | T
  T → T * F | F
  F → (E) | n

Some derivations:

E → T → F → n                          ∈ L(E)
E → E + T → T + T → ... → n + n        ∈ L(E)
E → E + T → ... → n + (n * n)          ∈ L(E)

Therefore: L(G) = { any arithmetic expression with * and + }, where n is a number (operand).

Adding productions E → E + T | E − T | T and T → T * F | T / F | T % F would give a language expressing all arithmetic expressions with *, ÷, +, −.

Example 3 — a simple programming language:

Program    → block "#"
block      → "{" stmt-List "}"
stmt-List  → statement ";" stmt-List | λ
statement  → if-stmt | while-stmt | read-stmt | write-stmt | assignment-stmt | block
...
read-stmt  → "read"
write-stmt → "write"

Non-terminals: {Program, block, stmt-List, statement, if-stmt, while-stmt, read-stmt, write-stmt, assignment-stmt}

Terminals: {"{", "}", "#", ";", "if", "while", "read", "write"}

Some derivations:

Program → block # → {stmt-list} # → {λ} # → {} #
Program → block # → {stmt-list} # → {read ; write ;} #

L(G) here is the set of all valid programs that can be written in this language. Each complete, valid program is one sentence in L(G). For real languages like C or Pascal, there are hundreds of productions — but the principle remains the same.

Algorithms for Derivation

There are two main derivation techniques:

Leftmost Derivation — replace the leftmost non-terminal at each step:

E → E + T → T + T → F + T → n + T → n + F → n + n

Rightmost Derivation — replace the rightmost non-terminal at each step:

E → E + T → E + F → E + n → T + n → F + n → n + n

Both arrive at the same final sentence regardless of the order used.

Example — grammar V → SR$, S → + | − | λ, R → .dN | dN.N, N → dN | λ:

Leftmost derivation of −ddd.d$:

V → SR$ → −R$ → −dN.N$ → −ddN.N$ → −dddN.N$ → −ddd.N$ → −ddd.dN$ → −ddd.d$

Rightmost derivation of −ddd.d$:

V → SR$ → SdN.N$ → SdN.dN$ → SdN.d$ → SddN.d$ → SdddN.d$ → Sddd.d$ → −ddd.d$

Which strategy does each parser type use?

  • Top-down parsers use leftmost derivation.
  • Bottom-up parsers use rightmost derivation (in reverse).

Important distinction: Leftmost/rightmost describe which non-terminal to replace at each step. Top-down/bottom-up describe the direction of the parse tree construction. These are separate concepts.

Derivation Trees

A Derivation Tree (parse tree) displays the derivation of a sentence in the language. Traversing the tree in order and recording only the leaves yields the sentence.

Example tree for −ddd.d from V → SR$, S → + | − | λ, R → .dN | dN.N, N → dN | λ (leftmost derivation):

          V
         / \
        S   R          $
        |   |
        −   d   N
                |
                d   N
                    |
                    d   N    .   N
                        |        \
                        λ      d   N
                                   |
                                   λ

Classes of Grammars

According to Chomsky, there are 4 classes of grammars (from least to most restrictive):

1. Unrestricted Grammars

No restrictions whatsoever beyond the definition that the left side contains at least one non-terminal. Not practical for compiler construction.

2. Context-Sensitive Grammars

For each production α → β, |α| ≤ |β| — the right-hand side is at least as long as the left-hand side. This means no λ-productions (A → λ) are allowed, since |λ| = 0 < |A|.

Why λ-productions matter for parsers: Parsers depend heavily on λ-productions to represent optional language constructs, empty statements, and terminate recursive rules. Example: statement-list → statement statement-list | λ. Without λ-productions, representing optional elements and recursive structures becomes very difficult.

3. Context-Free Grammar (CFG)

Each production has the form A → α, where A is a single non-terminal and α ∈ V*. This is the most important class — most programming language structures are context-free. We will mostly work with this class.

V* = all strings formed from V, including λ. So α can be:

  • A mix of terminals and non-terminals (e.g. aAbC)
  • Only terminals (e.g. abc)
  • Only non-terminals (e.g. ABC)
  • The empty string λ

4. Regular Grammar

Each production has the form A → aB or A → a, where A, B ∈ V_N and a ∈ V_T, with the exception S → λ. This is more restrictive than CFG because the right-hand side must be a single terminal, or a single terminal followed by a single non-terminal.

Subset relationship: Regular Grammars ⊂ Context-Free Grammars

  • Every regular grammar is also a context-free grammar.
  • Regular grammars are simpler and used by scanners.
  • Context-free grammars are more powerful and used by parsers.

Example:

A → aA     (terminal followed by non-terminal)
A → a      (single terminal)
→ L(G) = a⁺

Adding A → λ gives L(G) = a*
