## Formal Definition

A grammar **G = (V_N, V_T, S, P)** where:

1. **V_N** вҖ” a finite set of **non-terminals**.
2. **V_T** вҖ” a finite set of **terminals**.
3. **S вҲҲ V_N** вҖ” the **starting symbol** of the grammar.
4. **P** вҖ” a set of **production rules** that define the grammar structure.

> **Notes:**
> - `V_N вҲ© V_T = вҲ…` вҖ” the two sets are disjoint.
> - `V_N вҲӘ V_T = V` вҖ” together they form the full vocabulary.

**Notation conventions used throughout this chapter:**

- Uppercase letters A, B, вҖҰ, Z вҶ’ **non-terminals**
- Lowercase letters a, b, вҖҰ, z вҶ’ **terminals**
- Greek letters Оұ, ОІ, Оі, вҖҰ вҶ’ strings formed from V_N or V_T (i.e. from V)

For example, if `V_N = {S, A, B}` and `V_T = {0, 1}`:
```
Оұ = A11B
ОІ = S110B
Оі = 0010
```

## Productions

A **production** `Оұ вҶ’ ОІ` (read "alpha derives beta") is a rewriting rule that allows us to replace any occurrence of Оұ with ОІ in a string.

> The left-hand side Оұ must contain **at least one non-terminal** from V_N.

If we have the string `ОіОұПғ`, we can apply the production `Оұ вҶ’ ОІ` to obtain `ОіОІПғ`.

A **derivation** is a sequence of strings obtained by repeatedly applying productions:

```
ОұвӮҖ вҶ’ ОұвӮҒ вҶ’ ОұвӮӮ вҶ’ ... вҶ’ Оұn
```

Given a grammar G, **L(G)** is the set of all strings (the language) that can be generated by G.

**Example 1:**

```
G = ({S, B, C}, {a, b, c}, S, P)

Productions:
  S вҶ’ aSBC
  S вҶ’ abC
  CB вҶ’ BC
  bB вҶ’ bb
  bC вҶ’ bc
  cC вҶ’ cc
```

Some derivations:
```
S вҶ’ abC вҶ’ abc                           вҲҲ L(G)
S вҶ’ aSBC вҶ’ aabCBC вҶ’ aabBCC вҶ’ aabbCC вҶ’ aabbcC вҶ’ aabbcc   вҲҲ L(G)
S вҶ’ aSBC вҶ’ ... вҶ’ aaabbbccc              вҲҲ L(G)
```

Therefore: **L(G) = { aвҒҝbвҒҝcвҒҝ | n вүҘ 1 }**

> There is no exact method to find L(G) вҖ” the idea is to try different derivation paths and identify the pattern.

**Example 2:**

```
G = ({E, T, F}, {+, *, (, ), n}, E, P)

Productions:
  E вҶ’ E + T | T
  T вҶ’ T * F | F
  F вҶ’ (E) | n
```

Some derivations:
```
E вҶ’ T вҶ’ F вҶ’ n                          вҲҲ L(E)
E вҶ’ E + T вҶ’ T + T вҶ’ ... вҶ’ n + n        вҲҲ L(E)
E вҶ’ E + T вҶ’ ... вҶ’ n + (n * n)          вҲҲ L(E)
```

Therefore: **L(G) = { any arithmetic expression with * and + }**, where n is a number (operand).

> Adding productions `E вҶ’ E + T | E вҲ’ T | T` and `T вҶ’ T * F | T / F | T % F` would give a language expressing all arithmetic expressions with *, Г·, +, вҲ’.

**Example 3 вҖ” a simple programming language:**

```
Program    вҶ’ block "#"
block      вҶ’ "{" stmt-List "}"
stmt-List  вҶ’ statement ";" stmt-List | О»
statement  вҶ’ if-stmt | while-stmt | read-stmt | write-stmt | assignment-stmt | block
...
read-stmt  вҶ’ "read"
write-stmt вҶ’ "write"
```

Non-terminals: `{Program, block, stmt-List, statement, if-stmt, while-stmt, read-stmt, write-stmt, assignment-stmt}`

Terminals: `{"{", "}", "#", ";", "if", "while", "read", "write"}`

Some derivations:
```
Program вҶ’ block # вҶ’ {stmt-list} # вҶ’ {О»} # вҶ’ {} #
Program вҶ’ block # вҶ’ {stmt-list} # вҶ’ {read ; write ;} #
```

L(G) here is the **set of all valid programs** that can be written in this language. Each complete, valid program is one sentence in L(G). For real languages like C or Pascal, there are hundreds of productions вҖ” but the principle remains the same.

## Algorithms for Derivation

There are two main derivation techniques:

**Leftmost Derivation** вҖ” replace the **leftmost** non-terminal at each step:

```
E вҶ’ E + T вҶ’ T + T вҶ’ F + T вҶ’ n + T вҶ’ n + F вҶ’ n + n
```

**Rightmost Derivation** вҖ” replace the **rightmost** non-terminal at each step:

```
E вҶ’ E + T вҶ’ E + F вҶ’ E + n вҶ’ T + n вҶ’ F + n вҶ’ n + n
```

Both arrive at the same final sentence regardless of the order used.

**Example** вҖ” grammar `V вҶ’ SR$`, `S вҶ’ + | вҲ’ | О»`, `R вҶ’ .dN | dN.N`, `N вҶ’ dN | О»`:

Leftmost derivation of `вҲ’ddd.d$`:
```
V вҶ’ SR$ вҶ’ вҲ’R$ вҶ’ вҲ’dN.N$ вҶ’ вҲ’ddN.N$ вҶ’ вҲ’dddN.N$ вҶ’ вҲ’ddd.N$ вҶ’ вҲ’ddd.dN$ вҶ’ вҲ’ddd.d$
```

Rightmost derivation of `вҲ’ddd.d$`:
```
V вҶ’ SR$ вҶ’ SdN.N$ вҶ’ SdN.dN$ вҶ’ SdN.d$ вҶ’ SddN.d$ вҶ’ SdddN.d$ вҶ’ Sddd.d$ вҶ’ вҲ’ddd.d$
```

> **Which strategy does each parser type use?**
> - **Top-down parsers** use leftmost derivation.
> - **Bottom-up parsers** use rightmost derivation (in reverse).

> **Important distinction:** Leftmost/rightmost describe *which non-terminal to replace* at each step. Top-down/bottom-up describe the *direction of the parse tree construction*. These are separate concepts.

## Derivation Trees

A **Derivation Tree** (parse tree) displays the derivation of a sentence in the language. Traversing the tree in order and recording only the **leaves** yields the sentence.

Example tree for `вҲ’ddd.d` from `V вҶ’ SR$`, `S вҶ’ + | вҲ’ | О»`, `R вҶ’ .dN | dN.N`, `N вҶ’ dN | О»` (leftmost derivation):

```
          V
         / \
        S   R          $
        |   |
        вҲ’   d   N
                |
                d   N
                    |
                    d   N    .   N
                        |        \
                        О»      d   N
                                   |
                                   О»
```

## Classes of Grammars

According to **Chomsky**, there are 4 classes of grammars (from least to most restrictive):

**1. Unrestricted Grammars**

No restrictions whatsoever beyond the definition that the left side contains at least one non-terminal. Not practical for compiler construction.

**2. Context-Sensitive Grammars**

For each production `Оұ вҶ’ ОІ`, `|Оұ| вүӨ |ОІ|` вҖ” the right-hand side is at least as long as the left-hand side. This means **no О»-productions** (A вҶ’ О») are allowed, since |О»| = 0 < |A|.

> **Why О»-productions matter for parsers:** Parsers depend heavily on О»-productions to represent optional language constructs, empty statements, and terminate recursive rules. Example: `statement-list вҶ’ statement statement-list | О»`. Without О»-productions, representing optional elements and recursive structures becomes very difficult.

**3. Context-Free Grammar (CFG)**

Each production has the form `A вҶ’ Оұ`, where A is a **single non-terminal** and Оұ вҲҲ V*. This is the **most important class** вҖ” most programming language structures are context-free. We will mostly work with this class.

> V* = all strings formed from V, including О». So Оұ can be:
> - A mix of terminals and non-terminals (e.g. `aAbC`)
> - Only terminals (e.g. `abc`)
> - Only non-terminals (e.g. `ABC`)
> - The empty string О»

**4. Regular Grammar**

Each production has the form `A вҶ’ aB` or `A вҶ’ a`, where A, B вҲҲ V_N and a вҲҲ V_T, with the exception `S вҶ’ О»`. This is more restrictive than CFG because the right-hand side must be a single terminal, or a single terminal followed by a single non-terminal.

> **Subset relationship:** Regular Grammars вҠӮ Context-Free Grammars
> - Every regular grammar is also a context-free grammar.
> - Regular grammars are simpler and used by **scanners**.
> - Context-free grammars are more powerful and used by **parsers**.

Example:
```
A вҶ’ aA     (terminal followed by non-terminal)
A вҶ’ a      (single terminal)
вҶ’ L(G) = aвҒә

Adding A вҶ’ О» gives L(G) = a*
```
